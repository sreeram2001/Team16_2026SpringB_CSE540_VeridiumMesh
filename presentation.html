The Problem

  • Carbon credit markets lose $250M+ annually to fraud
  • Projects inflate issuance volumes, credits get double counted, some projects don't even exist
  • No centralized authority can be trusted to screen every single credit
  • Existing registries are opaque and lack real time verification

What We Need

A transparent, tamper proof system where every credit is screened by AI and permanently recorded on an immutable ledger before anyone can trade it.

Harpreet presents

Why Blockchain?

Immutability

Once a credit is recorded on Ethereum, nobody can alter or delete it. The audit trail is permanent.

Transparency

Anyone can verify the full history of any credit. No hidden modifications, no backdoors.

Smart Contract Enforcement

Business rules are enforced by code, not people. No human can bypass the dual approval requirement.

Digital Signatures

ECDSA signatures prove identity cryptographically. Only the key holder can transfer or retire their credits.

Smart Contract: 5 Blockchain Principles

1. ERC-721 NFTs

Every credit is a unique non-fungible token. Transfers and burns use the ERC-721 standard. MetaMask shows them as NFTs.

2. ecrecover Multi-Sig

Minting requires ECDSA signatures from both a registered developer and regulator. Contract recovers signers on chain.

3. Proof of Work

Every mint needs a nonce where keccak256(creditId + nonce) has top 8 bits = 0. About 256 attempts on average.

4. Merkle Tree

Each credit is a leaf in an on-chain tree. Inclusion proofs verified in O(log n) via OpenZeppelin MerkleProof.

5. Decentralization & Access Control

Admin can register multiple registrars, developers, and regulators independently. No single address controls all roles. Risk score >= 0.7 is rejected by the contract automatically.

AI Scoring Layer

Isolation Forest Model

  • Trained on Berkeley VROD dataset (5,700 real carbon credit projects)
  • Unsupervised anomaly detection
  • Score: 0.0 (safe) to 1.0 (suspicious)
  • Threshold: >= 0.7 is HIGH RISK and rejected

Four Features

  • R_ratio: tonnes / 50,000 (inflation vs peer average)
  • Vintage_Age: how old the project is
  • M_flag: 1 if historically risky type (Solar, REDD+, Wind)
  • T_flag: 1 if volume > 3x average

Low Risk Example

Cookstoves, 5,000 tonnes, recent vintage

Score: 0.25 ✓

High Risk Example

Solar, 600,000 tonnes, 18 year old vintage

Score: 0.85 ✗ Rejected

Score stored on chain as uint256 (float × 10,000). Example: 0.8451 → 8451

Limitations & Trade-offs

Limitation Impact Why We Made This Choice
In-memory pending queue Lost on server restart Simpler for demo; production would use PostgreSQL
Backend signs on behalf of developer Centralized trust for endorsement Hardhat accounts can't do browser signing for the contract call
Merkle tree recomputed fully on each mint O(n) gas cost grows with credits Simpler implementation; incremental updates would reduce gas
Low PoW difficulty (~256 hashes) Easy to mine Fast for demo; tunable for production
Local Hardhat node only No real consensus or persistence Free, fast, deterministic for development
Unsupervised ML model Catches anomalies, not confirmed fraud No labeled fraud dataset available for supervised learning

Future Improvements

Testnet Deployment

Deploy to Sepolia or Polygon Mumbai for real network behavior with actual consensus.

Persistent Storage

PostgreSQL for the pending queue so credits survive server restarts.

IPFS Documents

Store project documentation on IPFS and link the hash to each credit on chain.

Better ML Model

Supervised learning if labeled fraud data becomes available. Ensemble methods for higher accuracy.

Gas Optimization

Batch minting and incremental Merkle updates to reduce per-credit gas costs.

Privacy (zk-SNARKs)

Zero knowledge proofs so auditors can verify credits without seeing all details.
